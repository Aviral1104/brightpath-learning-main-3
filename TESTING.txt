1. TESTING FRAMEWORK OVERVIEW
================================================================================

Framework   : Vitest
Reason      : Vitest is the recommended testing framework for Vite-based projects.
              It uses the same configuration pipeline as Vite, resulting in faster
              test execution with zero additional configuration overhead.

Environment : jsdom
              Tests run in a simulated browser DOM environment (jsdom), allowing
              React components and browser APIs (window, document, localStorage,
              sessionStorage) to be tested without a real browser.

API         : Jest-compatible (describe, it, expect, beforeEach, afterEach, vi)
              Any developer familiar with Jest can write and read Vitest tests
              without learning a new API.

Config File : vitest.config.ts
Setup File  : src/test/setup.ts
Test Files  : src/**/*.test.ts | src/**/*.spec.tsx

--- 2.2 Focus Timer Logic ---

    The Focus Timer context (FocusTimerContext.tsx) was tested to verify:
    - Timer counts down correctly from 25:00 (Focus) and 5:00 (Break)
    - Phase switches from Focus → Break → Focus automatically
    - Session count increments after each completed focus session
    - Timer state persists across component remounts (context-level state)
    - Reset function correctly restores to initial Focus phase at 25:00

--- 2.3 Progress Calculation ---

    The useStudentProgress hook was tested to verify:
    - Percentage calculation: completedCount / totalSubchapters × 100
    - Edge case: denominator = 0 returns 0% (no divide-by-zero)
    - Confetti trigger only fires on genuine 0→100 transition, not on page load


================================================================================
3. INTEGRATION TESTING
================================================================================

Integration tests verify that multiple modules work together correctly.

--- 3.1 Firebase Auth + Firestore Profile ---

    Tested flow:
    1. User signs up via Firebase Authentication
    2. Auth trigger creates a Firestore document in /profiles/{uid}
    3. Role field (teacher / student / parent) correctly gates dashboard access
    4. Sign-out clears the session and redirects to /auth

--- 3.2 Firestore Security Rules ---

    The production Firestore security rules (firestore.rules) were validated
    using the Firebase Rules Playground in the Firebase Console:

    ✓ Unauthenticated read of /courses → DENIED
    ✓ Student attempting to write to /courses → DENIED
    ✓ Teacher writing to own course (teacher_id == uid) → ALLOWED
    ✓ Student creating own submission (student_id == uid) → ALLOWED
    ✓ Student marking own progress (studentId == uid) → ALLOWED

--- 3.3 Amazon Polly Text-to-Speech Integration ---

    End-to-end integration was verified using a Node.js test script:

    Command run:
        node --input-type=module test-polly.mjs

    Result:
        ✅ Amazon Polly working! Audio: 16028 bytes | audio/mpeg

    Verified:
    - AWS credentials authenticated successfully
    - SynthesizeSpeechCommand returned a valid MP3 audio stream
    - Audio blob URL created and played in browser via HTMLAudioElement
    - Module-level cache prevents duplicate API calls for the same text

--- 3.4 TTS API Quota Check (RapidAPI) ---

    During development, the original RapidAPI TTS quota was verified to be
    exhausted via a live HTTP test:

        Status : 429 Too Many Requests
        Reason : Monthly character quota exceeded (BASIC plan)

    This confirmed the need to migrate to Amazon Polly.
